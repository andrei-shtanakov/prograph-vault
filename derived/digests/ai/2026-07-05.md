## 2. MCP 2026-07-28 Release Candidate: a stateless core, MCP Apps, and Tasks — the biggest protocol revision since launch

**What happened:** The release candidate of the Model Context Protocol specification is out, with a target date of 2026-07-28 — the biggest revision since launch. The key parts: a stateless core, response caching, an extensions framework, server-rendered UI via MCP Apps, a reworked Tasks extension for long-running work, authorization closer to OAuth/OpenID Connect, and a formal deprecation policy. A ten-week window is given for Tier-1 SDKs and clients to validate the changes on real workloads.

**Architecture/Details:** Previously a remote MCP server required sticky sessions, a shared session store, and deep packet inspection at the gateway; now it can live behind an ordinary round-robin balancer, route traffic by the `Mcp-Method` header, and allow clients to cache `tools/list` for `ttlMs`. Extensions are identified by a reverse-DNS ID, live in separate `ext-*` repositories, and are versioned independently of the spec.

**My take:** This is exactly the class of news that, for me as an engineer, is more valuable than yet another model release, because a standard outlives any specific model. We discussed the upcoming MCP revision in previous digests as a roadmap — now it's a concrete RC, and the move from "session-based stateful" to "stateless + cache" is not cosmetics but a change of operating model: MCP servers finally scale like normal stateless HTTP services rather than a capricious stateful zoo. MCP Apps (server-rendered UI) and the Tasks extension are an admission that "tool call → response" is too narrow for real agentic workflows, where actions are long and need to be tracked. Separately pleasing is the formal deprecation policy: until now the ecosystem broke silently. I keep a sober skepticism: the extensions framework with reverse-DNS IDs and delegated maintainers sounds nice, but fragmentation is a real risk, and "Tier-1 SDKs will make it in 10 weeks" is a plan, not a fact. And the OAuth alignment, however correct it may be, will add pain for everyone who has already jury-rigged their own auth over the old spec. But the direction is right, and I'd already start building stateless compatibility into my MCP servers now.

## 3. Claude Tag: Anthropic turns a Slack bot into a permanent AI colleague and buries the old app on August 3

**What happened:** Anthropic launched Claude Tag — not a one-on-one assistant but @Claude as a full participant in a Slack channel: one model per channel, visible to everyone, accumulating context, taking initiative, and working asynchronously. In beta for Enterprise and Team. The legacy Claude for Slack app is being retired on August 3, 2026 — IT admins have 30 days to opt in and configure the new system, otherwise the switch happens automatically. Inside Anthropic they claim that 65% of the code of their product team is written by the internal version of Claude Tag.

**Architecture/Details:** Three differences from a chatbot: multiplayer (one @Claude per channel, anyone picks up the thread), memory accumulation per channel (no need to re-explain context), and an "ambient" mode — proactive updates, flagging of relevant info from connected tools, and reminders on stalled threads/tasks.

**My take:** Behind the boring phrase "Slack bot" hides an important product shift: the agent stops being a stateless function that gets called and becomes a stateful entity with memory and initiative, tied to a channel rather than to a user. The multiplayer model is the right engineering intuition: real work happens in shared channels, and "one Claude for all" beats "a personal assistant for everyone," because context lives where the conversation happens. The "65% of the product team's code" figure is a strong signal, but it's exactly the one I hold maximum skepticism toward: it's an internal, unauditable metric, and "code created by Claude" can mean anything from boilerplate autogeneration to genuine authorship; such metrics fit perfectly into the IPO narrative. As an engineer I'm more worried by the "ambient" mode: an agent that itself decides what I "need to know" and posts to a shared channel is both a convenience and a new class of risks (noise, context leakage between people, prompt injection via channel messages). And the "August 3 or auto-switch" deadline is aggressive: IT teams are being forced to migrate an agent with access to work correspondence in 30 days. Worth trying, but with explicit permission boundaries and ambient off until audited.
