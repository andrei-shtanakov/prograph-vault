## 1. Squidbleed: a one-line bug in Squid, alive for 29 years, found by AI

**What happened:** Researchers at Calif.io disclosed Squidbleed (CVE-2026-47729) — a Heartbleed-style vulnerability in the Squid proxy server that, in the default configuration, leaks internal process memory: other clients' cleartext HTTP requests along with Authorization headers and session tokens. The bug is a single line from a January 1997 commit that survived 29 years of releases, reviews, and independent audits. It was found not by hand but with the help of an Anthropic model (Claude Mythos Preview) as part of an initiative to find holes in open source.

**Technical details:** This is a heap over-read in the FTP gateway (FtpGateway.cc): the whitespace-skipping logic calls `strchr(w_space, '\0')` without checking that the current byte is not a NUL terminator, so `strchr` runs off the end of the buffer. Only scenarios with cleartext HTTP and TLS termination are affected — an HTTPS proxy relays traffic as an opaque CONNECT tunnel. It requires the proxy to be able to reach the attacker's FTP server (port 21). The fix is a NUL check before the call; it was merged into dev in April and into v7 in May, but Squid 7.6 effectively failed to ship it, and the real patch is promised in 7.7. The temporary measure is to disable FTP.

**My take:** The bug itself is niche: narrow exploitation conditions and declining FTP traffic make it not a "remote Armageddon." But the news is not about Squid, it's about exactly how it was found. The thesis "many eyes make all bugs shallow" is now definitively dead: a one-line error survived three decades of reviews and paid audits in one of the most widespread proxies on the internet — and yielded not to a human but to an LLM specifically trained on memory-safety patterns. This changes the economics of open-source auditing: machine analysis is starting to find what people systematically miss in C legacy. The practical takeaway is twofold — check your Squid configs and disable FTP if it's not needed, but strategically brace for a wave of similar findings in old C code, because someone is now finally reading it in full. And separately, it's unpleasant that "the fix exists" (7.6) and "the fix actually arrived" (7.7) are different events: check not the version number but the changelog.

## 2. Cisco SD-WAN under root: a zero-day exploited for two months before disclosure

**What happened:** On June 25, Mandiant (Google) disclosed the details of active exploitation of CVE-2026-20245 in Cisco Catalyst SD-WAN Manager (vManage), Controller (vSmart), and Validator (vBond). This is a command injection (CVSS 7.8): an authenticated attacker executes arbitrary commands with root privileges by uploading a specially crafted file. According to Mandiant, an unknown group exploited the hole as a zero-day for at least two months before public disclosure, attacking the SD-WAN infrastructure of a service provider.

**Technical details:** Through the tenant-upload feature in the CLI, a malicious CSV `evil_tenant.csv` was uploaded: the payload backed up configs, created a `troot` account with root privileges, after which the attacker switched to it via `su`. Throughout the operation, anti-forensic techniques were applied — selective deletion and restoration of modified configuration files. The fixes are in releases 20.9.9.2, 20.12.7.2, 20.15.x, 20.18.3.1, 26.1.1.2, and newer.

**My take:** "Authenticated, CVSS 7.8" sounds reassuring right up until you realize this is about the management plane of the orchestrator of the entire SD-WAN network — and admin accounts get phished out every day. This is again a story about the network's control center being the most tempting target, and two months of undetected presence with careful track-cleaning says this is not a script kiddie but a mature actor who understands vManage internals. What grabs me is precisely the vector: uploading "data" (a CSV into a tenant form) that turns into code execution — a classic failure of trust in input files where no one expects injection. For everyone operating Cisco SD-WAN: the patch is half the job, the other half is an audit for rogue accounts like `troot` and anomalies in config history, because for those who were already breached, the patch will clean nothing out.

**Novelty:** ★★★☆☆ | **Significance:** ★★★★☆ | **Impact:** ★★★★☆

## 3. Linkerd 2.20: the service mesh slims down 85% on control-plane memory

**What happened:** On June 23, Buoyant released Linkerd 2.20 — a new version of the cloud-native service mesh. The key point: engineers reworked the resource consumption of control-plane components and cut memory requirements by roughly 85% without sacrificing reliability or security guarantees. Plus, rate-limit-aware load balancing, automated trust anchor (root certificate) rotation, and support for Windows VMs outside Kubernetes appeared.

**Technical details:** Rate-limit awareness is built directly into the routing logic: traffic is dynamically steered away from overloaded upstream targets to preserve the platform's overall throughput (essentially load-shedding at the mesh level). A new trust-anchor rotation operator fully automates root-certificate updates with guardrails against downtime. Windows-VM support extends the mesh beyond Kubernetes clusters onto legacy infrastructure.

**My take:** 85% on memory is not cosmetics but a direct answer to the main argument against a service mesh: "it eats resources and adds overhead on every pod." Linkerd has for years held the line of "a small Rust data plane versus a bloated Istio," and reducing the control plane's appetite reinforces exactly that position — for large clusters, memory savings convert directly into money. I like the rate-limit-aware balancing conceptually: moving overload protection to the network layer rather than smearing it across every service is the right engineering decomposition. Automatic trust-anchor rotation closes the classic "the day the root cert expired and everything went down." One skeptical note: Windows VMs outside Kubernetes is about a bridge to legacy, and such features usually drag in the very complexity they were added for. But overall the release is about maturity, not hype, and that's exactly what you want from an infrastructure layer.

## 5. Phantom Gyp: the Miasma worm mutated and now strikes AI-IDE configs

**What happened:** On June 24, an attacker published malicious versions of 20 npm packages of the Leo Platform ecosystem in under three seconds — a new variant of the Miasma worm (the same one that infected Red Hat packages in early June). The main novelty is the Phantom Gyp technique: instead of the tracked preinstall/postinstall hooks, the code executes via a 157-byte `binding.gyp` file at the `npm install` stage, bypassing most install-script scanners. The payload steals secrets from GitHub Actions runners, cloud credential stores, package registries, and password managers, and exfiltrates them through the victim's own GitHub token. In parallel, the repository simonecorsi/mawesome was compromised: a force-push of a malicious commit and re-pointing of tags onto it.

**Technical details:** `binding.gyp` is a node-gyp build config, executed when installing native modules; moving the payload there bypasses both `--ignore-scripts` and most lifecycle-hook monitors. But the most alarming thing is that the variant injects persistent backdoor files directly into the project's repository, which trigger when a developer opens the project in an AI-assisted IDE. That is, a new class of persistence emerges through the configs of AI tools.

**My take:** I flagged the npm supply-chain theme last time (Miasma in Red Hat packages), but here there's a substantial update, for which the news is worth including: the technique has shifted to where defenders aren't looking. For all the recent months the industry chanted "set `--ignore-scripts`, monitor preinstall/postinstall" — and Phantom Gyp elegantly moves into `binding.gyp` and zeroes out exactly that measure. This is a vivid lesson that security through blocking a known vector always loses: the attacker simply finds the neighboring door. But what really worries me is the second thing — a backdoor activated by opening the project in an AI-IDE. We added a powerful new code executor to the work cycle (an agent in the editor), which automatically reads repository configs and acts on them — and it has already turned into a persistence vector. For everyone building CI/CD and working in an AI-IDE: pin and vendor dependencies, isolate installation in a sandbox, audit `binding.gyp` in third-party packages, and treat AI-assistant auto-run configs with extreme wariness. Localhost stopped being a trust boundary back at AutoJack — now `npm install` and opening a project in an editor also cease to be safe operations.
