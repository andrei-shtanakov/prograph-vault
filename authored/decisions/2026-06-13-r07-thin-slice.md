# R-07 thin slice: benchmark-aware routing in arbiter

> Date: 2026-06-13. Status: proposal (design note).
> Goal: close the "benchmark → routing" loop with a minimal change,
> without retraining the tree, without changing contracts, and without schema migrations.
> All references are to arbiter@`docs/r-06b-m4-update` (HEAD 2026-05-23).

## TL;DR

1. **The gap is one line:** the `benchmark_runs` table (db.rs:903) is written via `report_benchmark`→`insert_benchmark_run` (db.rs:757), but **no one reads it**. `route_task.rs`, `features.rs`, `policy/` do not touch it (grep comes back empty). The contract is built (M4 ✅), but the data leaks into the void.
2. **The slice = a post-inference re-rank by the benchmark score**, NOT a new feature in the 22-dimensional vector. It mirrors the already existing `PREFERRED_AGENT_BOOST = 0.1` pattern in `route_task.rs`. Zero retraining of the sklearn tree, zero change to `n_features`, zero migrations.
3. **Volume:** 1 new reader in `db.rs` (~15 lines) + ~25 lines in `route_task.rs` + 1 golden test. Behind the `ARBITER_BENCH_WEIGHT` flag (default 0.0 = off) — a safe A/B, with no regression when the table is empty.
4. **The real blocker is data, not code.** `benchmark_runs` currently has 0 rows. To run the first A/B without resurrecting Maestro: a one-off seed script from ready ATP results (atp-platform is hot, real runs exist) → `report_benchmark`. The full loop through `maestro benchmark` requires R-06b M5 (pending) — that is a separate step.

## 1. Where the decision is made today (as-is)

```
route_task.rs:
  build candidates (Vec<AgentInfo>)                    # route_task.rs:190
    → build_feature_vector(agent) -> [f64; 22]         # features.rs:123
       (f[9] = agent_success_rate from agent_stats,    # features.rs:174
        default 0.5; source — report_outcome)
    → evaluate_for_agents(tree, vectors)               # engine.rs:20
       -> Vec<(agent, PredictionResult)> sort by confidence desc
    → + PREFERRED_AGENT_BOOST to confidence             # route_task.rs (const:62)
    → invariant checks + cascade fallback → selection
```

The 22-dimensional vector (engine.rs:55-72) is entirely about the task + operational agent stats. **Benchmark performance is not reflected in any dimension.** The reranking hook is right at hand — `PREFERRED_AGENT_BOOST` already adjusts confidence before the selection.

## 2. The decision and why not the alternatives

| Option | What changes | Verdict |
|---------|-----------|---------|
| **B. Post-inference confidence re-rank** (chosen) | reader + ~25 lines in route_task | ✅ Thin, reversible, observable, A/B flag |
| A. A new 23rd feature in the tree | `n_features` 22→23 everywhere (Rust+JSON+contract), retrain sklearn, new goldens | ❌ Not thin; touches the model contract |
| A′. Mix the score into f[9] success_rate | conflates two signals in one feature | ❌ Muddies the semantics, silent regression of the operational stat |

Re-rank wins: the tree stays a black box as is, the benchmark enters as a separate, explicitly tunable axis, the effect is logged into `decision_path`, and the A/B is an env flag.

## 3. Concrete changes

### 3.1. Data reader — `arbiter-mcp/src/db.rs` (next to `get_agent_stats`, ~:409)

```rust
/// Latest benchmark score for an agent (most recent ts). None if no run.
/// Score clamped to [0,1] defensively — the contract does not guarantee the range
/// (report_benchmark-v1.schema.json: "score": {"type":"number"}).
pub fn get_latest_benchmark_score(&self, agent_id: &str) -> Result<Option<f64>> {
    let row = self.conn.query_row(
        "SELECT score FROM benchmark_runs WHERE agent_id = ?1 \
         ORDER BY ts DESC LIMIT 1",
        [agent_id],
        |r| r.get::<_, f64>(0),
    ).optional()?;                       // uses idx_benchmark_runs_agent_bench_ts (db.rs:920)
    Ok(row.map(|s| s.clamp(0.0, 1.0)))
}
```
> v1: "the latest score per agent". v2 (later): filter `benchmark_id` under the task's `task_type` + EWMA over several runs. For the slice — deliberately a single run.

### 3.2. Re-rank — `arbiter-mcp/src/tools/route_task.rs` (after `evaluate_for_agents`)

```rust
// next to PREFERRED_AGENT_BOOST (:62)
const DEFAULT_BENCH_WEIGHT: f64 = 0.0; // OFF by default — opt-in A/B

let bench_weight = std::env::var("ARBITER_BENCH_WEIGHT")
    .ok().and_then(|s| s.parse::<f64>().ok())
    .unwrap_or(DEFAULT_BENCH_WEIGHT);

if bench_weight > 0.0 {
    for (agent_id, pred) in ranked.iter_mut() {
        if let Some(score) = db.get_latest_benchmark_score(agent_id)? {
            // center on 0.5: no data / average agent = neutral (0 shift)
            let delta = bench_weight * (score - 0.5);
            pred.confidence = (pred.confidence + delta).clamp(0.0, 1.0);
            decision_path.push(format!(
                "bench_adjust[{agent_id}]: score={score:.3} Δ={delta:+.3}"));
        }
    }
    ranked.sort_by(|a, b| b.1.confidence
        .partial_cmp(&a.1.confidence).unwrap_or(std::cmp::Ordering::Equal));
}
```
> Idempotent to an empty table: no rows → no shift → behavior byte-for-byte as today. The shift is centered on 0.5 (like the default success_rate), so "no benchmark" = neutral, not a penalty.

### 3.3. Observability
`bench_adjust[...]` already lands in `decision_path` → is serialized into `decisions.decision_path` (db.rs:864) and into the obs span. Additionally: one field `bench_weight_applied: f64` in `RouteResult` for A/B analysis.

## 4. Where to get the data (this is the real blocker)

`benchmark_runs` = 0 rows. Two paths, in order of cost:

**Path 1 (recommended for the first A/B — decoupled from Maestro).**
A one-off seed script: take real ATP-eval outputs (atp-platform is hot, runs exist) → build a payload per `report_benchmark-v1.schema.json` → send it to the arbiter MCP `report_benchmark` (or a direct INSERT for the smoke). We get 3–6 rows (claude_code/codex_cli/aider × 1–2 benchmarks) → enable `ARBITER_BENCH_WEIGHT=0.15` → see the routing shift on the same features. **Tests the hypothesis in an evening, without touching Maestro.**

**Path 2 (the full loop).**
Close R-06b M5 — the CLI `maestro benchmark <id> --agent ...`. All the M4 wiring (`BenchmarkResult`→`report_benchmark`) is ready, but there is no entry point (pending in the roadmap). This is a separate piece of work in Maestro, not in this slice.

> Pragmatics: Path 1 gives the "routing really changes from benchmarks" signal at the lowest cost. If the signal is there — Path 2 and reviving Maestro are justified. If on real scores the route does not move meaningfully — that is an honest "kill signal" for the whole direction, obtained in a single evening.

## 5. Verify (part of the slice)

- **Golden test** (`arbiter-mcp/tests/`): two agents with an identical feature vector, different `benchmark_runs.score` → with `ARBITER_BENCH_WEIGHT>0` the agent with the higher score ranks first; with weight=0 / an empty table — the order does not change (regression guard).
- **Smoke on a live db** after seeding: `route_task` with the flag and without — diff in `decision_path` and in the selected agent.

## 6. What is NOT included (scope guard)

No: retraining the tree, changing `n_features` (stays 22), edits to `report_benchmark-v1.schema.json`, migrations, changes to the invariant rules, EWMA/multi-run aggregation, benchmark_id↔task_type mapping. All of that is v2, after the hypothesis is confirmed.

## Recommended actions

1. **arbiter:** add `get_latest_benchmark_score` (db.rs) + the re-rank block (route_task.rs) + a golden test. (~1 evening)
2. **arbiter/scripts:** a seed script from ATP results → `report_benchmark` (Path 1). (~1 evening)
3. **Run the A/B** `ARBITER_BENCH_WEIGHT` 0.0 vs 0.15 on a live db, record the shift in `_cowork_output/status/`.
4. **Decision based on the outcome:** signal exists → plan R-06b M5 (the full loop); no signal → close Maestro/arbiter as paused, justified by the data.
