# R-06b — what is this feature?

> **Status:** Implemented (2026-05-23) — see Maestro PRs #19/#20/#21 and arbiter PRs #11/#13/#14/#15
> **Date:** 2026-04-25
> **Supersedes:** the R-06b formulation in `_cowork_output/roadmap/ecosystem-roadmap.md:244` ("ATP verification via the Python SDK")
> **Related:** R-06a ✅ (CLI quick win, closed), R-07 (eval-driven routing, depends on this)
> **Addendum 2026-05-23:** §8 locks in the M4 decisions (channel choice + storage shape). Open Q #1 — resolved.

## TL;DR

The way R-06b is framed in the roadmap ("ATP verification via the SDK, closes the feedback loop in the Arbiter") is inaccurate. Once we dug into the SDK, it turned out that:

- The ATP SDK is built around "**ATP hands out tasks → the agent solves them → ATP scores them**".
- In other words, the SDK is a client for a **benchmark participant**, not a client for a validator.
- Under the name "R-06b ATP verification" the roadmap has fused three **different** features together. They need to be separated, one of them chosen, and the whole thing reformulated.

**The decision of this doc:** reformulate R-06b as **"Agent benchmarking via ATP"** (feature F2 below), not as "validation deepening".

---

## 1. What R-06b delivers to the product — three features under one name

### F1. Validation deepening
- **What:** Maestro writes tests inside task specs; ATP runs them and returns a structured report (rather than just an exit code 0/1/2).
- **Who initiates:** Maestro (as in R-06a).
- **SDK needed?** No — this is a **different** ATP API. The SDK is not for validation; for validation we already have the `atp run` CLI (R-06a).
- **Score:** R-06a already closed the basic scenario. Deepening is a new feature of a different nature (HTTP API + structured output), not "via the SDK".
- **Verdict:** not R-06b. If it is needed — a separate task with a different dependency.

### F2. Agent benchmarking ⭐
- **What:** Maestro runs its spawned agents (claude_code, codex_cli, aider) through an external ATP benchmark. The benchmark scores flow into the Arbiter as a new routing signal: "`claude_code = 0.72` on swe-bench-mini, `codex_cli = 0.65` → route swe-bench-like tasks to claude_code".
- **Who initiates:** Maestro (or CI), periodically.
- **SDK needed?** **Yes** — exactly the use case the SDK was built for (`ATPClient.start_run` → iterate tasks → submit).
- **The goal of "structured feedback into the Arbiter" is preserved**, but the meaning of the feedback **changes**: not "spec X passed/failed", but "agent Y scored Z on benchmark B".
- **Verdict:** this is the reformulated R-06b.

### F3. Maestro-as-participant (export)
- **What:** Maestro registers itself in ATP as an "agent"; external users test Maestro-as-orchestrator through ATP benchmarks.
- **SDK needed?** Yes, the reverse direction: Maestro answers ATP tasks via DAG execution.
- **Verdict:** out of scope for R-06b. This is at the "Maestro as a product" level, not the feedback loop.

### Why this matters

A vs B (SDK-validator vs SDK-spawner-adapter), which I proposed earlier, is about the **integration method** for an unspecified feature. Without F1/F2/F3 the A/B axis is meaningless. First the feature, then the method.

## 2. Sequence diagram (F2 — the chosen feature)

```
  CI/Operator         Maestro              ATP SDK            ATP Server         Arbiter
       │                 │                    │                   │                 │
       │─benchmark cmd──▶│                    │                   │                 │
       │                 │─ATPClient.start_run▶│─POST /runs──────▶│                 │
       │                 │                    │◀───run_id, tasks──│                 │
       │                 │◀─run iterator──────│                   │                 │
       │                 │                    │                   │                 │
       │      ┌──── loop over tasks ─────┐    │                   │                 │
       │      │ spawner.spawn(claude_code)│    │                   │                 │
       │      │ wait for agent response   │    │                   │                 │
       │      │ run.submit(response)      │───POST /submit───────▶│                 │
       │      └───────────────────────────┘    │                   │                 │
       │                 │                    │◀──final_score─────│                 │
       │                 │─report_outcome     │                   │                 │
       │                 │   (benchmark_score, agent, run_id)─────────────────────▶│
       │                 │                    │                   │                 │
```

What is immediately visible here:
- **Maestro = executor** (not validator), ATP = the task requester.
- The score arrives **per-agent per-benchmark**, not per-task.
- This is a **new feedback channel** into the Arbiter, parallel to the ordinary `report_outcome` from the Maestro task lifecycle.
- No `validation_cmd: atp-sdk://...` — that was the wrong analogy.

## 3. What happens to the notion of ValidationResult

- **`ValidationResult`** — **stays** as is. Used in the R-06a CLI path and for the ordinary `validation_cmd`. F2 does not overlap.
- **`BenchmarkResult`** — a **new** signal type:
  ```python
  class BenchmarkResult(BaseModel):
      run_id: str          # ATP run identifier
      benchmark_id: str
      agent_id: str        # Maestro agent type that was benched
      score: float
      score_components: dict[str, float]   # per-metric breakdown
      per_task: list[BenchmarkTaskResult]  # for drill-down
      total_tokens: int | None
      total_cost_usd: float | None
      duration_seconds: float
      ts: datetime
  ```
- **Arbiter feedback** — extend, don't break: either a new MCP tool `report_benchmark`, or project it into the existing `report_outcome` via a special `task_id = f"benchmark:{run_id}"`. Decide later (see open questions).

## 5. Thin slice (M1) — what exactly to do first

**Goal:** validate the API wiring without depending on a live ATP server and without auth.

```python
# tests/test_benchmark_runner.py
class MockATPClient:
    def start_run(self, benchmark_id, agent_name) -> MockRun:
        return MockRun(tasks=[
            {"metadata": {"task_index": 0}, "prompt": "fix bug X"},
            {"metadata": {"task_index": 1}, "prompt": "fix bug Y"},
        ])

class MockSpawner:
    def respond(self, prompt: str) -> str:
        return f"resolved: {prompt}"

def test_runner_iterates_tasks_and_submits():
    runner = BenchmarkRunner(MockATPClient(), MockSpawner())
    result = runner.run(benchmark_id="swe-mini", agent_name="claude_code")
    assert result.benchmark_id == "swe-mini"
    assert len(result.per_task) == 2
    assert MockATPClient.submitted == [(0, "resolved: fix bug X"),
                                       (1, "resolved: fix bug Y")]
```

**Files for M1:**
- `maestro/benchmark/__init__.py`
- `maestro/benchmark/runner.py` — `BenchmarkRunner`
- `maestro/benchmark/models.py` — `BenchmarkResult`, `BenchmarkTaskResult`
- `tests/test_benchmark_runner.py` — mock-only

**What we do NOT do in M1:**
- Real `ATPClient` (network calls)
- Auth flow
- Real spawners (claude_code subprocess)
- Arbiter feedback
- CLI command

## 8. M4 design resolutions (addendum, 2026-05-23)

> **Source:** spec-runner side input in response to a question from the Maestro developer.
> **Status:** **Recommended** — pending Maestro/arbiter sign-off.
> **Scope:** this section closes Open Q #1 and locks in the storage shape for M4. Open Q #2–#5 remain open.

### 8.1. Feedback channel — `report_benchmark` as a new tool (resolves Open Q #1)

Chosen option (a): **a new MCP tool `report_benchmark`** in arbiter, not a channel-in-`report_outcome`.

**Why not (b) "squeeze it into `report_outcome` via `task_id = f"benchmark:{run_id}"`":**
- String-typed routing by a `task_id` prefix is a **hidden contract**. Six months from now some code will appear that parses the prefix in unexpected places (metrics, logs, dashboards), and renaming the prefix will no longer be possible without coordinating all consumers.
- `report_outcome` semantically is a **per-task outcome** from the ordinary task lifecycle. `BenchmarkResult` is **per-agent-per-benchmark**, a different axiom. Mixing them = breaking the invariant of one of the two.
- The tool name documents the channel. `report_benchmark` reads clearly at once; `report_outcome` with a magic prefix does not.

**Cost:** one Rust change in arbiter (~tool + ~table + ~tests). Reversible.

### 8.2. What arbiter does with `BenchmarkResult` — **persist-only**

Chosen **option 1**: `report_benchmark` writes to a new `benchmark_runs` table and returns a structured ack. No automatic updates to `agents`, no read API in M4. R-07 is a separate feature, responsible for inference, and it will add the reads it needs itself.

**Why not option 2 (persist + update `agents`):**
Premature commitment to **aggregation**. Which exact score should be written to `agents` — mean? EWMA? per-task-type? confidence interval? The moment a concrete number lands in a column, it becomes a **new arbiter routing contract** — and the DT inference in R-07 may well want something completely different (recency-weighted, percentile, per-task-class, threshold with N≥k runs). Better to choose the representation **where** it is consumed, not to freeze one of these answers in M4.

**Why not option 3 (persist + `get_benchmark_scores`):**
Doubling the API surface **without a consumer**. R-07 is not written — the read shape is a guess. A naive `get_benchmark_scores(agent_id?, benchmark_id?)` almost certainly will not fit the real DT inference (which needs filters by `task_type` + a time window + a per-metric threshold, not "give me everything for the agent"). We will add it under actual need — an additive minor.

**Advantages of option 1:**
- Minimal diff in arbiter Rust: 1 table + 1 tool, fully reversible.
- Clean separation of milestones: M4 = "data into arbiter", R-07 = "arbiter uses it". If they are mixed, M4 sprawls and R-07 becomes cosmetic.
- Visibility does not suffer: you can debug with SQL or a temporary CLI helper on the arbiter side — without tying an external MCP API to it.
- Reversibility: adding the tool/columns later = additive minor. Tearing it out = a breaking change to the Rust API for Maestro.

### 8.3. Return shape — not `{recorded: bool}`, but an ack with idempotency

Free now, expensive to add later:

```rust
struct ReportBenchmarkAck {
    recorded: bool,
    run_id: String,           // echo for logging on the Maestro side
    dedup: DedupStatus,       // "inserted" | "duplicate"
    server_ts: DateTime<Utc>, // when arbiter accepted the record
}
```

**Why it matters:**
- **Idempotency.** Maestro may retry on a network failure. Without `dedup`, Maestro either pours in duplicates or builds its own idempotency-key logic.
- **Server timestamp.** Separate "when the benchmark finished" (`BenchmarkResult.ts`) from "when arbiter accepted the record" (server_ts). Computing recency in R-07 needs server_ts; otherwise clock skew on Maestro will poison the ranking.

PK for dedup: `(run_id, agent_id)` — `run_id` is unique in ATP, `agent_id` is optional in case several agents are benched in parallel under one `run_id` (not the current scenario, but cheap insurance).

### 8.4. Schema `benchmark_runs` — lock it as a contract right away

The table in arbiter is internal, but its columns are **already** a contract between Maestro (writer) and R-07 (future reader). Stylistic ask: alongside M4, drop a short spec into `_cowork_output/contracts/benchmark-runs-schema.md` (1 page: columns, types, PK, retention policy). This does **not** require a JSON Schema file — we do not serialize it as JSON on the wire, the format lives inside Rust. Just a markdown table + a brief description of the fields.

Minimal set of columns (for the write path; R-07 will refine the read shape later):

| Column | Type | Source |
|---|---|---|
| `run_id` | `text PK part` | `BenchmarkResult.run_id` |
| `agent_id` | `text PK part` | `BenchmarkResult.agent_id` |
| `benchmark_id` | `text` | `BenchmarkResult.benchmark_id` |
| `score` | `real` | `BenchmarkResult.score` |
| `score_components` | `jsonb` | `BenchmarkResult.score_components` |
| `per_task` | `jsonb` | `BenchmarkResult.per_task` (for drill-down) |
| `total_tokens` | `bigint?` | optional |
| `total_cost_usd` | `numeric?` | optional |
| `duration_seconds` | `real` | |
| `benchmark_ts` | `timestamptz` | `BenchmarkResult.ts` (when the run finished) |
| `server_ts` | `timestamptz` | NOW() at insert |

R-07 will add indices/derived columns on top of this if needed.
