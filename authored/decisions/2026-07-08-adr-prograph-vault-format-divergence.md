# ADR: prograph ↔ prograph-vault format divergence + sync automation

- **Status:** accepted — staging-via-promoter approach (2026-07-08); the code-side `export_root` change (P0a) is not yet implemented.
- **Affects:** `prograph` (generator), `prograph-vault`/ecosystem-kb (store), indirectly `robin-runtime` (KB consumer).

## TL;DR

1. prograph writes all output hard-wired under `<root>/.prograph/` (`prograph/prograph/paths.py:36-53`); it cannot be redirected into `derived/graph/` by config — only by code.
2. The sync `.prograph/projects/*.md → derived/projects/*.md` is **manual** today (a git commit "re-sync from fresh prograph output"); cards are byte-identical, so it is an external `cp`, not a prograph feature.
3. **Name collision on `contracts/`:** prograph emits generated cards; the vault's `derived/contracts/` holds verbatim schema snapshots (`type: contract-snapshot`, `sha256`) written by a separate kb skill. One folder — two producers and formats.
4. `graph.db` (the graph itself, ~20 tables) does not reach the KB (gitignored binary) — the KB keeps only the rendered cards, losing edge/drift queryability.
5. prograph itself has **no `.git`** historically — a canon generator with no version history (governance gap). *(Update 2026-07-08: prograph now has git + remote.)*

## Decision

**Staging-via-promoter is adopted** (not direct prograph writes into the KB).

The fork "`export_root` → `derived/graph/` directly vs via staging" was analyzed (below). Conclusion: **staging is more effective** — and not a new staging dir, but a promote step on top of prograph's already-existing second phase (`.prograph/`).

1. **Split the namespace.** prograph output → `derived/graph/{projects,contracts,index.md}`. The kb-skill contract snapshots stay in `derived/contracts/`. The collision disappears; "tool owns derived/graph" becomes clean.
2. **KB-write ownership belongs to the promoter, not prograph.** prograph writes only to a **repo-local staging** (`.prograph/`, later config `export_root`). A separate promoter (`authored/skills/promote-prograph.sh`) moves staging → `derived/graph/` atomically, with `rsync --delete`, a drift gate, and a scoped commit. prograph stays self-contained and does not know the KB path.
3. **Ship a machine-readable graph** (`graph.json`) next to the MD — so Robin/consumers read edges/drift without the binary and without re-running prograph.
4. **Stamp provenance** (`source_commit`, `generator_version`, `promoted_at`, `drift_findings_at_promote`) — the promoter writes `derived/graph/PROVENANCE.md`; provenance in card frontmatter is also desirable (P1).

### Fork: direct vs staging (why staging)

| Criterion | Direct → `derived/graph/` | Staging + promoter |
|---|---|---|
| Coupling | prograph hard-wired to another repo's path | prograph self-contained; promoter knows the layout |
| Atomicity | crash mid-run → broken KB, Robin reads a half-write | atomic promote; KB always consistent |
| Quality gate | tool output lands in canon unreviewed | drift/diff gate between generate and commit |
| Deleting stale | delete cards directly in KB (may hit adjacent) | `rsync --delete`, scoped to `derived/graph/` |
| Who commits | a commit is still needed → "direct" only removes the `cp` | promoter owns generate→check→commit |
| Moving parts | fewer (1 step) | more (a promote script) — but lives in kb-tooling |

Three facts vote for staging: the manual deletion of `proctor-a.md` (`c275d5e`) — direct without a controlled `--delete` leaves exactly that; **Robin is a live read-only consumer** of `derived/` (needs atomicity); the ecosystem is building gated governance (steward gate-check) — unreviewed tool output into an SSOT is against its own vector. The only downside of staging is one extra script, and it is small. **Direct is justified** for a single writer, low stakes, zero-friction; not for an SSOT under an autonomous consumer.

## Format divergence (summary)

| prograph output (`.prograph/`) | In the vault (`derived/`) | Match |
|---|---|---|
| `projects/*.md` | `derived/projects/*.md` | byte-identical, but **manual** sync |
| `contracts/*.md` (generated by schema URL) | `derived/contracts/*.md` (snapshot + sha256, kb skill) | different producer/format, **name collision** |
| `index.md` (generated) | root `index.md` (authored) | not synced |
| `graph.db` | — | not in KB (gitignored binary) |

The KB boundary intent: tools write `derived/`, humans write `authored/` (`prograph-vault/CLAUDE.md`). The boundary is **violated by `derived/contracts/`** — it is under `derived/` but hand-produced.

## Code recommendations for prograph

### P0a — prograph: configurable export root (repo-local staging)

Paths are hard-wired to `.prograph/` (`paths.py:36-53`) and MD dirs are inseparable from `db_path`.

- `paths.py`: add optional `export_root: Path | None = None`; resolve `projects_md_dir`/`contracts_md_dir`/`index_md_path` from `export_root` when set, else from `.prograph/`. **Do not touch** `db_path`/`lock_path`/`config_path` — DB stays in `.prograph/`.
- `config.py`: add `read_export_root(config_path)` (`[output] export_root = ".prograph/graph"`, relative to root), mirroring `read_auto_export` (`config.py:13-24`).
- `cli.py`: `--out-dir` on `index` and `export-md` (`cli.py:147,266`), priority over config; thread into every `PrographPaths(..., export_root=...)` (`cli.py:109,156,213,282,317,418`).

**Important:** `export_root` targets **staging** (repo-local, e.g. `.prograph/graph`), NOT the KB. prograph must not know the `prograph-vault` path. Until P0a lands, the promoter works against the current `.prograph/{projects,contracts,index.md}` layout — prograph code does not block it. Full brief: [[2026-07-08-prograph-export-root-task]].

### P0b — prograph-vault: promoter owns the write into `derived/graph/`

`authored/skills/promote-prograph.sh` (ready): `prograph index --export-md` → drift gate (`prograph drift --json`) → `rsync --delete` staging → `derived/graph/{projects,contracts,index.md}` → provenance stamp → scoped commit of `derived/graph` only. Flags: `--dry-run`, `--fail-on-drift`, `--no-reindex`, `--no-commit`.

### P1 — JSON graph export into the KB

`graph.db` is a gitignored binary, so the KB loses queryability. Add a serializer next to `render_index` (`export/render.py:275`): `graph.json` (nodes + edges with kind/attrs + drift + snapshot meta) into `export_root`. So Robin and dispatcher read the graph without the binary.

### P1 — provenance in card frontmatter

change_log already captures `git_commit`. Thread `source_commit` and `generator_version` into `_frontmatter` (`export/render.py:345`) so the vault side auto-detects staleness instead of trusting the date.

### P2 — drift as a CI gate

If `prograph drift` has no exit code, add `--exit-code` (non-zero when findings of the chosen `--kind` exist). Fits the steward/spec-runner gate philosophy.

### P2 — version prograph itself

A canon generator must have history. `git init` + remote. *(Done 2026-07-08.)*

## Alternatives

- **Keep the manual `cp` sync** — cheap, but drifts between runs and does not scale to jobs/Robin. Rejected (the `proctor-a.md` incident already shows the failure mode).
- **Store `graph.db` in the KB** — closes the queryability gap, but a binary in a git KB (merges, size, diff-unreadable). Worse than `graph.json`.
- **Move snapshot contracts out of `derived/contracts` into `authored/`** — also removes the collision, but snapshots are derived; the `derived/graph` split is cleaner.
