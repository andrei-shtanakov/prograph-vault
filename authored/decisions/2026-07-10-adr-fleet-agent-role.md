# ADR: Fleet agent — роль, дом, сенсоры и писатель `derived/fleet/`

**Status:** Proposed (2026-07-10) · **Surfaced by:** обсуждение «единого центра
управления» 2026-07-10; аудит доразбора `_cowork_output/` (там же).

## Context

Экосистема имеет мониторинг (dispatcher), Q&A по знаниям (Robin), governance
спеков (steward), исполнение (Maestro/spec-runner), но не имеет роли, которая
*сводит состояние флота репозиториев и планирует действия*. Обсуждались два
варианта дома: расширение robin-runtime (ломает его read-only-инвариант) и
новый репо (третье место для cross-cutting при незакрытом overlap-триаже).
Команда исходной машины разрешила вопрос фактом: тулинг выведен из
`_cowork_output/devtools/` в отдельный репо `devtools` (2026-07-10) — «малый
workspace-репо» (манифест + bootstrap + тулинг), README которого сам называет
его кандидатом на дом fleet-агента.

## Decision

1. **Роль.** Fleet-агент — управляющий флотом: наблюдает состояние всех репо
   workspace и инициирует действия. Границы (не дублировать):
   dispatcher *смотрит* (read-only дашборд runtime-артефактов), Robin
   *отвечает* (Q&A по KB), steward *гейтит* (spec governance), fleet-агент
   *управляет* (сводка → отчёт → предложения → спеки).
2. **Дом — репо `devtools`.** Конституция роли — `devtools/CLAUDE.md`.
   ROBIN-SPEC не ревизуется: Robin остаётся справочным.
3. **Действия только косвенные.** Fleet-агент не пишет в чужие репо напрямую —
   только ветка → PR → ревью; задачи развития оформляет `tasks.md`-спеками
   для spec-runner в репо-владельце изменений.
4. **Сенсоров два, намеренно:** `github-checker snapshot` (текущее состояние:
   ветки, ahead/behind, dirty, PRs/issues/alerts; владелец возможности —
   github-checker) и `devtools/recent_changes.py` (дельта за период,
   stdlib-only). Не сливать: у них разные потребители и зависимости.
5. **Писатель `derived/fleet/` (правка §2 конституции KB).** Третьим
   автоматическим писателем под `derived/` регистрируется fleet-агент
   (`devtools/fleet_report.py`): владеет `derived/fleet/` — датированные
   отчёты о состоянии флота, append-only, НЕ регенерируемые (исторические
   снимки, как journal — ADR 2026-07-06). Доставка отчётов — PR-ом
   (в отличие от prograph/kb-save: прямых коммитов у fleet-агента нет
   нигде, включая KB). Каждый отчёт несёт `host` — чьи локальные клоны
   описывает.

## Consequences

- Единая точка управления достигается одним клоном devtools; Robin, dispatcher
  и steward не меняют своих инвариантов.
- `derived/fleet/` появляется в директорной карте KB; kb-curator при
  фрешнес-аудите обрабатывает его как journal (архивирование старых отчётов,
  не удаление).
- Отчёты с разных машин различимы по `host` и не конфликтуют.
- Альтернатива «выделить workspace-манифест в отдельный нейтральный репо»
  остаётся дешёвым выходом при росте команды; фиксируется как отклонённая
  сейчас, не навсегда.

## Rejected

- **Дом в robin-runtime** — требует ревизии read-only-спеки Robin ради роли,
  которой достаточно devtools; смешивает «отвечает» и «управляет».
- **Новый пустой репо под агента** — третий источник дрейфа cross-cutting
  знаний при уже существующих vault и devtools.
