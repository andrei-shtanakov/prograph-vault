### New protocol risk (P0)

`arbiter#9`: the real-arbiter smoke run R-03 on the Maestro side (against the macos-arm64 binary `7e6de56`) revealed that the `route_task` response does not contain `metadata.decision_id`. The Maestro side (`coordination/routing._extract_decision_id`) and `FakeArbiterClient` expect this field. Symptom: arbiter ASSIGNs and writes `decisions.id = 1` to its SQLite, but Maestro stores `arbiter_decision_id = NULL`. Consequence: `_try_report_outcome` behaves as it would for static-routed; the decision-id guard on `reset_for_retry_atomic` does not fire; the authoritative retry / abandon flow is unobservable in real runs. **Maestro's 1118 tests are green** because `FakeArbiterClient` always returns `decision_id`. The **contract freeze** on DTO `861534e` (see `arbiter/TODO.md`) must now be revisited: either this is a bug in the arbiter implementation, or the Maestro side is misinterpreting the contract.
