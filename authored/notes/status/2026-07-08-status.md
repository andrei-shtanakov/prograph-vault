## Explicit misalignments

**1. Legacy name `proctor-a` vs `arbiter` (main finding).**
`arbiter` apparently grew out of an earlier component called `proctor-a`. The rename was left incomplete:

- **dispatcher** registers **two collectors at once**: the current `ArbiterCollector` (name=`arbiter`, detects the `arbiter-core` folder + reads `arbiter.db`) **and** the legacy `ProctorCollector` (name=`proctor-a`, detects `config/proctor.yaml` + `data/state.db`). No repo in the set produces `config/proctor.yaml` anymore, so `ProctorCollector` **will never find anything** — dead code in the `COLLECTORS` registry. dispatcher's README still lists `proctor-a` as a monitored target too.
- **Maestro** still has `proctor-a-scheduler.yaml` and `proctor-a-phase2.yaml` + 51 mentions of `proctor`; **dispatcher** has 77 mentions (collector, plans, tests).

Low risk (ArbiterCollector functionally covers arbiter), but it's accumulating terminology debt: confuses new readers and leaves a non-functional collector live in the dashboard.

**2. steward outside the link graph** (see "Who's lagging") — 0 mentions of `arbiter`/`proctor`, zero dependencies; wired into neither contracts nor orchestration.

**What's NOT wrong (checked and confirmed):** no drift in benchmark/observability schemas, no MCP protocol-version mismatch between Maestro↔arbiter, no uncommitted changes, no divergence from upstream. Contract discipline in the core is healthy.
