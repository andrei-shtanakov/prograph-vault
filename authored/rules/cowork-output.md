# `_cowork_output/` — конвенция

После KB-миграции (ADR 2026-07-05, канон живёт в `prograph-vault/`) `_cowork_output/` —
**только dev-scratch / staging**: зона, где артефакт *рождается* до повышения (graduation)
в реальный репо. Ничего постоянно-канонического здесь не живёт. Рантайп и shipped-код её
не читают (см. root `CLAUDE.md`, [[repo-boundaries]]).

## 1. Главное правило — graduate, don't duplicate
- Черновик, ставший каноном, **переносится** (не копируется) в KB `authored/` (через PR)
  или в репо-владелец. После этого ему выставляется `graduated_to: <путь>`, а исходник
  архивируется: он становится **замороженным указателем** и больше не редактируется —
  правки идут в канонический экземпляр.
- **Разрешение конфликта:** если копия в `_cowork_output/` и её канон в KB/репо разошлись —
  **канон побеждает по определению**: копию в `_cowork_output/` архивируем/выкидываем, назад
  никогда не мержим. Источник истины ровно один.

## 2. Целевая структура (лёгкая)
| Папка | Что кладём | Живёт долго? |
|---|---|---|
| `README.md` | короткий указатель на это правило | да |
| `drafts/` | одиночные черновики `YYYY-MM-DD-<kebab>.md`, ещё не канон | нет (до graduation) |
| `workstreams/<slug>/` | многофайловый набор под инициативу; в каждом свой `README.md` | нет (до завершения) |
| `devtools/` | dev-only тулинг (не шипается, рантайм не читает) | да |
| `_archive/YYYY/` | завершённый/устаревший скретч, сохранён для истории | да (пассивно) |

Папки-по-типам (`status/`, `decisions/`, `roadmap/`, `plans/`, `diagrams/`, `contracts/`)
**устарели как места назначения** — эти типы теперь у KB (`authored/decisions`,
`authored/notes`, `derived/contracts`). Новое сюда не заводим.

## 3. Frontmatter — только для graduation-кандидатов
`drafts/` держим низкофрикционным — scratch не бюрократизируем. Frontmatter несёт только
`.md`, который является кандидатом на повышение:

```yaml
---
title: <короткий>
status: draft | graduated | superseded
graduated_to: <путь в KB/репо>   # при повышении → файл становится замороженным указателем
owner: <кто>
updated: YYYY-MM-DD
---
```

## 4. Кто и что пишет
- **Cowork / TPM** (read-only на репо): пишет **только** в `_cowork_output/`; код репо — никогда.
- **Dev-сессия** (внутри подпроекта): основной вывод — в свой репо; корневой `_cowork_output/`
  только для **кросс-репного** черновика. Репо-локальный скретч → `<repo>/_cowork_output/`
  или `<repo>/docs/`.
- Любой репо-локальный `<repo>/_cowork_output/` **обязан** быть в `.gitignore` этого репо —
  иначе dev-scratch уедет клонирующим.

## 5. Git
- Мусор не коммитим: `.DS_Store`, `*.swp`, `*.kate-swp`, `__pycache__/` — в `.gitignore`.
- Корневой `_cowork_output/`: зонтичный репо без remote, весь durable-канон и так уезжает в
  реальные репо. Рекомендация: трекать максимум `devtools/` (+ этот `README.md`-указатель),
  повышенные черновики не трекать. Точный объём трекинга — решение владельца (в процессе).

## 6. Enforcement (важнее текста)
Текст снижает риск, но не enforce'ит. Цель (Stage 3, ср. `prograph-vault/CLAUDE.md:§9`):
gate-check, который падает, если (a) отредактирован файл с `graduated_to:`, либо (b) контент
дублируется между `_cowork_output/` и KB. Этот чек — реальный «забор» против drift'а, ради
которого правило и написано. См. [[git-workflow]].
