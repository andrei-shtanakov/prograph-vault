# Границы репозиториев

Экосистема `all_ai_orchestrators` — полирепо: каждый подпроект — отдельный git-репо.
Правило действует для любого агента/человека, работающего в подпроекте.

1. **Один агент — один репо.** Правь только тот репо, в котором запущен (его git-корень).
   Все соседние репо под `all_ai_orchestrators/` — READ-ONLY reference.
2. **Нужна правка у соседа → стоп.** Не редактируй чужой репо. Запиши handoff:
   кросс-проектное — сюда, в `authored/notes/YYYY-MM-DD-<kebab>.md`; черновик —
   `_cowork_output/`. Дальше делай только свою сторону изменения.
3. **Контракты — вендорингом внутрь.** Кросс-репный контракт потребляется пиненой копией
   внутри своего репо (см. root CLAUDE.md, ADR-ECO-003), а не импортом из пути соседа и не
   ссылкой в `_cowork_output/`. Авторитет контракта — в репо-владельце.
4. **Общее знание — read-only.** KB (`prograph-vault/authored/`) и `_cowork_output/`
   читаются из подпроекта, но не повод править код соседа.
5. **Enforcement > текста.** Это правило снижает риск, но не enforce'ит. Реальные рычаги:
   (a) запуск с рабочей директорией = папка подпроекта; (b) gate-check в CI, роняющий PR,
   который трогает пути вне репо; (c) `.claude/settings.local.json` с deny на запись вне
   корня. Правило — слой 1.

## Как ссылаться из репо
CLAUDE.md подпроекта держит короткий scope-блок (свой путь + remote + соседи) и ссылку
сюда. Не копировать это правило целиком в репо — только операционный дайджест.
