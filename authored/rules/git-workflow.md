# Git workflow

Действует для репозиториев **со своим remote**. Согласовано с полирепо-механикой
(`authored/notes/2026-05-25-polyrepo-workflow-setup.md`: не submodules, гетерогенные
ветки, `--ff-only`).

## Основной цикл
1. **Никаких прямых коммитов в дефолтную ветку** (`master`/`main`). Каждое изменение — на
   feature-ветке `<type>/<slug>` (`feat/`, `fix/`, `docs/`, `chore/`).
2. Push ветки → открыть PR (`gh pr create`). В описании: что, зачем, какие контракты затронуты.
3. **После открытия PR — ревью GitHub Copilot:**
   - забрать замечания (`gh pr view <n> --comments`, при необходимости `gh api
     repos/<owner>/<repo>/pulls/<n>/comments`);
   - каждое **проанализировать**: валидное → фикс-коммит в ту же ветку; невалидное/
     неприменимое → ответить с обоснованием, **не применять вслепую**;
   - итерировать, пока не останется открытых actionable-замечаний (или все закрыты с
     обоснованием).
4. **Не мержить.** PR передаётся человеку — мерж делает пользователь.
5. **После мержа пользователем:** `git switch <default> && git pull --ff-only`; удалить
   влитую локальную ветку (`git branch -d <branch>`); `git fetch --prune`; убрать прочие
   влитые ветки.

## Ограничения
- Никогда не force-push в общие ветки.
- Не трогать другие репо (см. `repo-boundaries.md`).
- `pull` только `--ff-only`; грязные/detached/без-upstream ветки пропускать.

## Исключения
- **Репо без remote** (`spec-runner-tasks`, корневой зонтик): только локальные коммиты,
  PR некуда — цикл выше неприменим.
- **Внешний upstream** (`sdd-framework` на `git.epam.com`, владелец не ты): PR идёт
  владельцу апстрима, **self-merge невозможен** — следуй их ревью, шаг 4–5 адаптируется.
- **Папки без `.git`** — правило неприменимо.
